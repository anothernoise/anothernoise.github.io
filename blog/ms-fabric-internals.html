Understanding MS Fabric Internals

📅 Published ⏱️ 8 min read

Microsoft Fabric represents a unified analytics platform that brings together data engineering, data warehousing, analytics, and AI workloads under a single, integrated experience. Understanding its internals is critical for architects designing modern lakehouse solutions.

What is MS Fabric?

MS Fabric is a Software-as-a-Service (SaaS) analytics platform built on Azure cloud infrastructure. It consolidates five major workloads:

  • Data Engineering: ETL/ELT pipelines using Spark and Delta Lake
  • Data Warehouse: SQL-based analytics on structured data
  • Lakehouse: Open format data lake supporting both unstructured and structured analytics
  • Real-Time Analytics: Event streaming and KQL-based time-series analysis
  • Power BI: Business intelligence and visualization layers

Core Architecture

Fabric's architecture revolves around a few fundamental concepts:

1. Workspaces & Capacity

Fabric organizes all resources within Workspaces, similar to folders or projects. Each workspace is associated with a Capacity—a reserved pool of compute and storage resources measured in Fabric Units (FU).

  • A workspace can contain multiple items: lakehouses, warehouses, pipelines, and semantic models
  • Capacity is shared across all workspaces assigned to it
  • Bursting is possible up to 4x capacity for short-duration spikes

2. Lakehouses: The Central Data Hub

The Lakehouse is Fabric's most fundamental data construct, combining the benefits of data lakes and warehouses:

  • Open Format Storage: Built on Delta Lake format, enabling compatibility with Apache Spark, Python, and SQL
  • Dual Endpoints: Each lakehouse exposes both a Lake endpoint (Spark/Notebooks) and SQL endpoint (T-SQL queries)
  • Automatic Indexing: Fabric creates hidden indexes on parquet files for query optimization
  • Versioning & Time Travel: Delta Lake provides ACID transactions and historical data access

Key Insight: The lakehouse SQL endpoint creates implicit views over Delta tables, making Spark-written data immediately queryable via SQL without format conversion.

3. Data Organization: Tables & Shortcuts

Within a lakehouse, data is organized into two zones:

  • Managed Tables: Data stored within the lakehouse's default storage account
  • External Tables: Data stored outside the lakehouse, referenced via shortcuts (similar to symbolic links)
  • Shortcuts: Enable cross-workspace and cross-cloud data federation

4. OneLake: Unified Storage

Fabric's storage layer, OneLake, provides:

  • Automatic data serialization to Parquet format for optimization
  • Built-in compression and columnar storage
  • Shortcut-based federation across workspaces
  • Compliance and governance at the storage layer

Compute Execution

Spark Pools & Notebooks

When you execute notebooks or Spark jobs in Fabric:

  • Fabric provisions serverless Spark clusters on-demand
  • Clusters auto-scale based on workload and shut down when idle
  • Execution is metered in Fabric Units; you pay for actual compute time
  • Data locality is optimized: Spark workers co-locate with OneLake data

SQL Compute (Warehouse & Lakehouse SQL)

SQL workloads run on dedicated SQL compute engines:

  • Warehouses use MPP (Massively Parallel Processing) architecture
  • Lakehouse SQL endpoints are read-only optimized views over Delta tables
  • Query pushdown: Filters and projections are applied at the storage layer
  • Caching is automatic; frequently accessed data is kept in memory

Data Flow & Pipelines

Fabric provides Data Factory for orchestrating ETL workflows:

  • Dataflows Gen2: Cloud-based data transformation using Power Query (M language)
  • Spark Jobs: Custom PySpark or Scala jobs for complex transformations
  • Pipeline Activities: Orchestrate copying, transforming, and loading data
  • Scheduling: Built-in scheduling with trigger-based and time-based execution

Governance & Metadata

Semantic Models & Power BI Integration

Fabric seamlessly integrates with Power BI:

  • Create semantic models (formerly datasets) directly from lakehouse tables
  • Models support DirectQuery and Import modes
  • Row-Level Security (RLS) can be applied at the model level

Purview Integration & Lineage

Fabric integrates with Microsoft Purview for governance:

  • Automatic data lineage tracking across pipelines and transformations
  • Data classification and sensitivity labeling
  • Access reviews and audit logging

Partitioning & Transaction Isolation

Fabric leverages Delta Lake's ACID transaction model with sophisticated partitioning strategies:

  • Partition Pruning: Queries automatically prune partitions based on filters, reducing I/O by 80%+
  • Z-Ordering: Multi-dimensional clustering of data for faster sorted access
  • Liquid Clustering: Dynamic, self-optimizing clustering without explicit partition schemas
  • Snapshot Isolation: Readers never see uncommitted writes; writers don't block readers
  • Conflict Resolution: Optimistic concurrency control with automatic retries on conflicts

Query Execution & Optimization

SQL Query Path

When you submit a SQL query to a lakehouse SQL endpoint:

  1. Parser: T-SQL syntax validation and transformation
  2. Planner: Cost-based optimizer creates execution plan using statistics
  3. Rewriter: Applies rule-based optimizations (predicate pushdown, join reordering, aggregate elimination)
  4. Executor: Distributes plan to worker nodes; results streamed to client
  5. Cache Layer: Result set cached for identical queries (15-min TTL by default)

Fabric's SQL engine uses Vectorized Query Execution, processing data in batches rather than row-by-row, yielding 10-100x speedups on analytics workloads.

Spark Execution Model

Spark jobs in Fabric follow a resilient distributed dataset (RDD) execution model:

  • DAG Execution: Lazy evaluation—transformations build a DAG; only actions trigger compute
  • Stage Optimization: Spark groups operations into stages; stages execute in parallel when dependencies allow
  • Task Distribution: Each partition in an RDD maps to a task; tasks distribute across worker cores
  • Fault Tolerance: Lineage tracking enables recomputation on node failures; no expensive checkpointing needed

Fabric APIs & SDK

Fabric exposes several programmatic interfaces for integrations:

  • REST APIs: CRUD operations on items, metadata, and capacity management
  • Python SDK: fabric-sdk for automation, workspace management, and item deployment
  • Azure SDK: Identity and RBAC integration for enterprise governance
  • JDBC/ODBC Drivers: SQL endpoint connectivity from BI tools (Tableau, Looker, etc.)
  • Synapse Spark APIs: Direct PySpark and Scala job submission with workspace isolation

Multi-Cloud & Hybrid Capabilities

Fabric supports federation across cloud boundaries:

  • AWS S3 Shortcuts: Query S3 data as if it were in OneLake without copying
  • Google Cloud Storage Integration: Federated queries across GCS and OneLake
  • On-Premises Data Gateway: Secure tunneling for on-prem SQL Server, Oracle, and Teradata
  • Cross-Cloud Lineage: Automatic lineage tracking across cloud boundaries for compliance

Advanced Security & Compliance

Encryption & Key Management

  • At-Rest Encryption: AES-256 encryption for all data in OneLake; customer-managed keys (CMK) supported
  • In-Transit Encryption: TLS 1.2+ for all network communication
  • Column-Level Encryption: Encrypt sensitive columns at ingest; queries decrypt on-the-fly with proper credentials

Access Control

  • Role-Based Access Control (RBAC): Workspace admin, member, contributor, viewer roles
  • Item-Level Permissions: Granular read/edit/delete controls on individual lakehouses and semantic models
  • Row-Level Security (RLS): Dynamic row filtering in semantic models based on user identity
  • Service Principals: App-owns-data authentication for unattended automation

Audit & Compliance

  • Activity Logging: All item creations, modifications, and deletions logged with user context
  • Query Auditing: SQL and Spark query execution logged with resource consumption
  • Purview Integration: Automated data classification, lineage tracking, and access reviews
  • Compliance Certifications: SOC 2 Type II, ISO 27001, HIPAA, FedRAMP compliance built-in

Capacity Metrics & Throttling

Fabric's metering engine tracks Fabric Units (FU) in real-time:

  • Compute FU: Charged per vCore-second of SQL and Spark execution
  • Storage FU: Charged per GB-month for data stored in OneLake
  • Background Operations: Automatic optimization jobs (vacuuming, compaction) metered separately
  • Throttling Policy: When capacity exhausted, queries queued with configurable max wait time (default 5 min)
  • Workload Management: Assign priorities (High/Normal/Low) to queries; high-priority queries preempt low-priority ones

Deep Dive: Fabric uses exponential backoff with jitter for queue management. When a query is throttled, its retry interval grows exponentially (1s → 2s → 4s...) with randomized jitter to prevent thundering herd on capacity release.

Advanced Performance Tuning

Statistics & Query Optimization

  • Fabric automatically collects histograms on table columns after ingestion
  • The optimizer uses these statistics to estimate row counts and choose optimal join orders
  • Stale statistics can degrade plan quality; refresh with ANALYZE TABLE after major data changes
  • Use EXPLAIN ANALYZE to see actual vs. estimated row counts and identify misestimations

Index-Free Performance

Unlike traditional databases, Fabric doesn't require manual index creation:

  • Parquet files are self-describing; column statistics enable predicate pushdown
  • File pruning based on min/max values eliminates scanning irrelevant files
  • Bloom filters on critical columns speed up equality and IN predicates
  • If certain queries remain slow, use materialized views to pre-compute aggregates

Join Strategies

  • Broadcast Join: For small tables <256MB, broadcast to all workers (default threshold)
  • Sort-Merge Join: For large tables; both sides sorted on join key, then merged in parallel
  • Hash Join: Default for equijoins; partitions both tables on join key, then local joins on workers

Performance Tuning in Fabric

Caching & Query Optimization

  • Result Caching: Query results are cached for identical queries
  • Columnar Compression: Parquet encoding reduces data transfer
  • Partitioning: Organizing data by date or category improves pruning
  • Statistics: Fabric automatically collects column statistics for query planning

Capacity Planning

Fabric uses a Capacity Utilization Metric to track resource consumption:

  • Throttling: When capacity is exhausted, queries are queued
  • Workload Management: You can prioritize critical workloads over exploratory queries
  • Auto-Pause: Warehouses can pause during periods of inactivity

Best Practices for Fabric Architectures

1. Data Modeling

  • Use dimensional modeling (star schemas) for analytics queries
  • Denormalize fact tables to reduce joins and improve query performance
  • Implement a medallion architecture: Bronze (raw), Silver (cleaned), Gold (analytics-ready)

2. Lakehouse Design

  • Organize data into logical lakehouses by domain or business unit
  • Use shortcuts to create virtual hierarchies across lakehouses
  • Keep raw data immutable in the Bronze layer for audit trails

3. Pipeline Optimization

  • Use Dataflows Gen2 for standard transformations; Spark for complex logic
  • Implement incremental loading to minimize full scans
  • Chain pipelines efficiently: avoid unnecessary re-materialization

4. Cost Management

  • Right-size your capacity: monitor Fabric Unit usage over time
  • Use pause/resume on warehouses to control costs during off-hours
  • Implement query caching to avoid redundant compute

Key Internals Takeaways

MS Fabric's internal architecture is built on several core principles that drive its capabilities and performance:

  1. Unified Storage (OneLake): Single source of truth with Delta format for all data types, enabling ACID transactions and time travel
  2. Serverless Compute: Auto-scaling Spark and SQL engines eliminate provisioning friction and enable true pay-per-use economics
  3. Data Federation: Query across clouds and on-prem via shortcuts without data movement, reducing ETL complexity
  4. Automatic Optimization: Statistics, caching, and intelligent query planning require minimal manual tuning
  5. Integrated Governance: Security, compliance, and lineage baked into the platform from day one; no bolt-on security
  6. Capacity-Based Metering: Transparent cost model based on actual resource consumption, not seat-based licensing

Conclusion

MS Fabric's internals—from OneLake storage to serverless Spark compute, from semantic models to governance—are architected to democratize analytics. Understanding these internals allows you to design scalable, cost-efficient, and maintainable data platforms that serve diverse analytical and AI workloads under a unified SaaS umbrella.

Whether you're migrating from legacy data warehouses or building greenfield lakehouse solutions, Fabric's internal architecture provides the flexibility and power to support modern data strategies.
